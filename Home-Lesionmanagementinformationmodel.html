Lesion management information model

The clinical data model used by the AIR module (Fig. 1) makes use of two types of FHIR resources that capture the findings of AI algorithms, and allow the input of the radiologist who is assessing them.

  • AIRBodyStructure profile: a specialization of the ILS BodyStructure profile, represents the lesion of normal tissue that is being assessed by the algorithm. Such modeling allows follow up of the finding through references to its representation as a BodyStructure. The attributes are the anatomical location (.location), the refinement of that location (.locationQualifier), and the type of lession(.morphology). Each use case profile described in this document has a binding to a value set with specific anatomical structures (.location) and a value set with types of lesions (.morphology) that can be observed in that modality/body region. LocationQualifier intends to be bound to a unique value set across multiple use cases.

  • AIRObservationRoot: this profile intermediates between the lesion (BodyStructure) through .focus and the following individual findings through .hasmember. It has a fixed .code:"http://snomed.info/sct"|"404684003"|"Clinical finding", and a .value.codeableconcept and .bodysite that match AIRBodyStructure.morphology and AIRBodystructure.location respectively and binding to the same value sets. It's used in the Bone fracture RX use case and in Lung nodule use case. It also holds the references to other resource types:

    • Patient: represents the person from who the images were taken.
    • ImagingStudy: provides information on a DICOM imaging study.
    • Device: represents an instance of the algorithm that outputs the findings in the study.
    • ServiceRequest: provides the record of an order for a service to be performed that would result in an imaging study.
    • Practitioner: represents the radiologist that interacts with the AIR system accepting, rejecting, or enriching the findings.

  • AIRObservationQuantitative captures the quantitative measurements of the lesion. It has a .code. that defines what is being meassured, bound to a unique value set across multiple use cases: AIRQuantitativeObservationValueSet . And a .ValueQuantity with binding to UCUM units or .valueRatio.

  • Qualitative measurements are captured in a specific profile for each type of feature. The .code element defines the type of feature observed with a fixed value, and .valueCodeableConcept stores the value within that domain with a binding to a specific value set. Two example profiles from a lung nodule use case in thorax CT scan are included in this implementation guide as a reference: AIRObservationSolidity and AIRObservationMargins (Fig. 1)
